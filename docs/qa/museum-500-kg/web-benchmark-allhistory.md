# MUSEUM-500-KG-WEB-00｜AllHistory 关系图谱基准研究与现有差距审计

## 1. 目的与结论

本文件不是对 AllHistory 视觉样式的复制说明，而是提炼其“关系图谱、AB路径、谱系、时空联动”背后的产品结构，并与 Museum-Codex 当前实现进行逐项对照。

结论：Museum-Codex 已解决“默认显示全部艺术家的圆环”和治理文案占据儿童首屏的问题，但目前仍是**艺术家单一节点类型上的策展比较浏览器**，不是完整的关系知识图谱。当前 258 位艺术家仅有 60 条正式关系，且全部为 C 级策展比较；Batch 02–05 扩展内容时没有新增历史关系层。500 位艺术家全部上线后必须重建关系数据模型、研究流程、路径算法和交互界面，不能继续在现有三条 lane 上堆功能。

- 官方产品说明支持的功能标记为“已核验”；
- 二级来源一致描述的交互标记为“可参考但需原型验证”；

- 不复制其图标、配色、布局、文案、数据或视觉资产。

## 3. AllHistory 值得借鉴的结构

### 3.1 搜索先于绘图

用户先提出一个实体或问题，再进入局部知识群落。系统不是先把整个数据库投射到屏幕上。Museum-Codex 必须保留“默认0节点”的正确方向，但下一步要让搜索支持艺术家之外的作品、展览、运动、机构、地点、材料、技法和文献。

### 3.2 人、事、物混合的异构图谱

官方说明强调世界上的“人、事、物”关联。对艺术馆而言，艺术家之间经常没有直接私人关系，但可以通过作品、展览、机构、艺术运动、地点、出版物、教师、赞助人和收藏事件形成可解释连接。只画艺术家节点会使绝大多数真实关系丢失。

### 3.3 图谱、路径、谱系是不同任务

- **图谱**：围绕当前实体动态展开局部邻域；
- **路径**：回答两个实体如何连接；
- **谱系**：表达师承、学院、工作室、团体、运动或技法传播的方向性结构。

三个任务不能共用一个没有语义的力导向画布，也不能仅靠切换筛选器冒充不同模式。

### 3.4 点击节点后重新聚焦

点击任意节点后，以该节点为中心继续探索，并保留访问历史、返回、固定和折叠。这比“一次展示某位艺术家的12个邻居”更接近持续发现知识的体验。

### 3.5 AB路径是解释工具，不只是最短线

AB路径的价值在于发现两实体之间的可解释链条。Museum-Codex 不能只按 hop 数给出最短路径，因为“巴黎”“油画”“现代艺术”“博物馆”等高连接度节点会制造无意义捷径。路径必须考虑关系类型、证据、时间一致性、节点泛化程度和路径多样性。

### 3.6 时间与空间是关系的语境

关系发生于何时、何地，决定它是直接接触、同场事件、间接传播、后世影响还是纯策展比较。关系图谱应与时间线和地图互相跳转，而不是把地点仅作为筛选标签。

## 4. 当前 Museum-Codex 实现

### 4.1 已完成的正确基础

- `focused_relation_lanes_v1`；
- 默认图节点数为0；
- 先选择艺术家；
- focus 初始邻居最多12，每 lane 最多4，扩展最多20；
- DOM + SVG，不依赖 WebGL；
- 节点和标签重叠门禁；
- 关系有“意味着什么／不意味着什么”；
- 完整艺术家列表和关系表作为文字等价；
- 算法边禁止公开；
- 儿童主叙事与证据层分离。

这些约束必须保留，不因500位规模而恢复全局大图。

### 4.2 数据层的根本不足

当前 `relationships.json` 的60条边全部是：

- Level C；
- `shared_subject`、`shared_material`、`shared_technique`；
- 无方向；
- `historical_relationship_strength=null`；
- 明确声明不代表相识、影响、师承或传播。

因此当前所谓“艺术家关系”实质上是**作品元数据驱动的策展比较**。它是有价值的观察工具，但不应继续承担“人物关系图谱”的名称和全部任务。

### 4.3 当前路径系统的不足

现有 path contract：

- 运行图主要由艺术家和艺术家关系组成；
- historical/context/comparison 三种模式虽在合同中存在，但公开关系只有C级；
- 最短路径使用 deterministic bidirectional BFS；
- 最多3条候选路径、最多6 hops；
- 首要排序是 hop count；
- `composite_weight=false`；
- 没有 generic hub penalty；
- 没有异构实体类型转换约束。

这意味着当前 AB路径无法表达“艺术家→展览→艺术家”“艺术家→学院→教师”“作品→收藏事件→机构”“艺术家→出版物→评论家”等知识链。

### 4.4 名称与信息架构问题

在3.0.0前应明确区分：

- **策展比较**：当前三类C级关系；
- **历史关系**：有直接来源的师承、合作、通信、家庭、赞助等；
- **语境连接**：通过事件、机构、地点、运动、文献等中间实体形成。

当前关系页不得暗示60条C级比较已经等同于艺术史关系网络。

## 5. 借鉴矩阵

| AllHistory方向 | Museum-Codex采用 | 不采用／需改造 |
|---|---|---|
| 搜索人、事、物 | 扩展为多实体统一搜索 | 不假定任意实体都有数据 |
| 点击节点持续探索 | click-to-recenter、expand、collapse、pin、history | 不无限展开造成视觉失控 |
| 图谱／路径／谱系 | 建立独立模式和数据投影 | 不把同一画布换标题冒充三种任务 |
| AB路径 | 多条可解释 typed paths | 不仅按最少 hop；不允许算法推断冒充事实 |
| 关系分类 | 历史、事件语境、策展比较三级分离 | 不照搬政治史中的敌对、婚姻等类别到所有艺术家 |
| 时空地图联动 | 关系边和事件可跳转时间／地点 | 不从机构所在地推断创作地点 |
| 动态视觉 | 局部、渐进、可恢复探索 | 不复制AllHistory视觉资产或一次显示全图 |
| AI知识图谱宣传 | 可使用规则化研究和算法排序 | 不声称AI生成的边为历史事实 |

## 6. 产品原则

1. **问题驱动，不是节点驱动。**
2. **异构实体，不是艺术家单一图。**
3. **历史事实、语境连接、策展比较严格分层。**
4. **每条公开关系必须 Claim → Evidence → Source。**
5. **路径是解释，不是图论炫技。**
6. **泛化枢纽必须惩罚，避免伪捷径。**
7. **时间和空间是边的属性与中间实体。**
8. **儿童层先解释“为什么连在一起”，研究层再展示证据。**
9. **没有证据时显示空状态，不为图好看造关系。**
10. **500位艺术家和5,000件作品上线后再以完整语料重构。**
11. **3.0.0部署前必须经过用户人工候选审阅。**
12. **不记录查询历史，不使用用户定位，不引入外部运行时图数据。**
